# ПРИНЦИПИ ДОКУМЕНТАЦІЇ - cross_pharm_market_analysis

> **Версія:** 1.3 | **Оновлено:** 10.02.2026

---

## 1. ГОЛОВНИЙ ПРИНЦИП

**Проект cross_pharm_market_analysis — це ШАБЛОН для мульти-ринкового аналізу субституції.**

Документація описує:
- **Патерни** — як організовані дані та процеси
- **Методологію** — як проводити аналіз
- **Концепції** — бізнес-логіку та рішення

Документація НЕ містить:
- Конкретних числових результатів
- Специфічних ID препаратів/аптек
- Жорстко закодованих параметрів

---

## 2. АУДИТОРІЯ ДОКУМЕНТАЦІЇ

| Аудиторія | Потреби |
|-----------|---------|
| **Розробники** | Розуміння архітектури, швидкий онбординг |
| **AI агенти** | Контекст для роботи з кодом |
| **Аналітики** | Методологія, формули, інтерпретація |
| **Замовники** | Бізнес-логіка без технічних деталей |

---

## 3. ПРАВИЛА ДОКУМЕНТУВАННЯ

### 3.1. Не дублювати код

❌ **Неправильно:**
```markdown
Параметри аналізу:
- CRITICAL_THRESHOLD = 0.40
- MIN_MARKETS = 5
```

✅ **Правильно:**
```markdown
Параметри аналізу визначені в `project_core/did_config/`
```

### 3.2. Посилатися на джерела

| Тип інформації | Джерело посилання |
|----------------|-------------------|
| Пороги класифікації | `project_core/did_config/` |
| Шляхи до даних | `project_core/data_config/` |
| Конкретні формули | Відповідний скрипт |
| Специфічні результати | Версійований документ |

### 3.3. Версіонування специфіки

Коли потрібно задокументувати конкретну реалізацію:

```
docs/{folder}/versions/
    └── {DD_MM_YY}_version_{type}.md
```

## 5. КОМПАКТНІСТЬ

### Принципи

1. **Одна ідея — один абзац** (максимум 3-4 речення)
2. **Таблиці замість списків** для структурованих даних
3. **ASCII-схеми** для візуалізації процесів
4. **Код-блоки** тільки для прикладів формул/структур

### Рекомендовані обсяги

| Тип документа | Рядків |
|---------------|--------|
| AI rules | 100-150 |
| Технічний опис етапу | 150-250 |
| Конфіг опис | 50-100 |

## 7. ОНОВЛЕННЯ ДОКУМЕНТАЦІЇ

### Коли оновлювати

- Після зміни логіки в скриптах
- Після додавання нових етапів
- Після зміни структури даних
- При виявленні помилок в документації

### Що оновлювати

1. **Версію** документа (X.Y → X.Y+1 для minor, X+1.0 для major)
2. **Дату** оновлення
3. **WORK_HISTORY.md** — запис про зміни

### Чого НЕ робити

- Не видаляти історичну інформацію без архівації
- Не змінювати документацію без узгодження
- Не дублювати інформацію між документами
